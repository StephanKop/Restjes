# Internationalization (i18n)

The platform is locale-aware. **Source of truth is Dutch (`nl`)**; English (`en-US`) is the first translation. The system is designed so adding a third language (e.g. `fr`, `de`) is a mechanical copy-paste, not a refactor.

---

## Architecture at a glance

```
packages/i18n/                     Shared message catalogs + config
├── src/
│   ├── config.ts                  locales, defaultLocale, localeMeta, helpers
│   └── index.ts                   re-exports JSON catalogs as named consts
└── messages/
    ├── nl/
    │   ├── common.json
    │   ├── footer.json
    │   └── profile.json
    └── en/
        ├── common.json
        ├── footer.json
        └── profile.json

apps/web/src/i18n/
├── request.ts                     next-intl server config — reads cookie, picks locale, loads messages
└── actions.ts                     server action: setLocaleAction(locale)

apps/web/src/components/LanguageSwitcher.tsx   web switcher (calls server action)

apps/mobile/lib/i18n.tsx                       React Context + i18n-js runtime
apps/mobile/components/LanguageSwitcher.tsx    mobile switcher
```

### Web stack: `next-intl`
- Cookie-based locale detection (`kc_locale`) with browser `Accept-Language` fallback.
- Server components use `await getTranslations('namespace')`.
- Client components use `useTranslations('namespace')` inside a `NextIntlClientProvider` wrapping the whole app (set up in `apps/web/src/app/layout.tsx`).
- Language switching = server action writes cookie → `revalidatePath('/', 'layout')` → whole tree re-renders.

### Mobile stack: `i18n-js` + `expo-localization`
- Initial locale = device language (from `expo-localization`) falling back to `nl`.
- Persisted to `expo-secure-store` under the key `kc_locale`.
- Exposed via `<LocaleProvider>` in `apps/mobile/app/_layout.tsx`.
- Use the `useTranslation()` hook in screens: `const { t } = useTranslation()`, then `t('profile.title')`.

## Using translations

### Web — server component (preferred)
```tsx
import { getTranslations } from 'next-intl/server'

export default async function Page() {
  const t = await getTranslations('footer')
  return <h1>{t('navHeading')}</h1>
}
```

### Web — client component
```tsx
'use client'
import { useTranslations } from 'next-intl'

export function MyWidget() {
  const t = useTranslations('common')
  return <button>{t('actions.save')}</button>
}
```

### Mobile
```tsx
import { useTranslation } from '../../lib/i18n'

export default function MyScreen() {
  const { t } = useTranslation()
  return <Text>{t('profile.title')}</Text>
}
```

### Interpolation
JSON:
```json
{ "copyright": "© {year} Kliekjesclub.nl" }
```
Usage:
```tsx
t('copyright', { year: new Date().getFullYear() })
```

Both `next-intl` and `i18n-js` use the `{placeholder}` syntax — keys written once work on both platforms.

## Adding a new translation key

1. Pick (or create) the right namespace file — e.g. `packages/i18n/messages/nl/profile.json`.
2. Add the key to **every locale** in that namespace. If you don't have the English copy yet, write Dutch under `en` temporarily — the build won't break, just the UI will fall back.
3. Use the key via the `t()` helper in your component.

> Missing-key behavior: `next-intl` throws in development, shows the key in production. `i18n-js` has `enableFallback: true` so it falls back to the default locale (Dutch).

---

## Adding a new namespace

1. Create `packages/i18n/messages/{nl,en}/<name>.json`.
2. Export it from `packages/i18n/src/index.ts`:
   ```ts
   export { default as messagesNlSomething } from '../messages/nl/something.json'
   export { default as messagesEnSomething } from '../messages/en/something.json'
   ```
3. Register it in both runtimes:
   - Web: `apps/web/src/i18n/request.ts` → add to the `messagesByLocale` map.
   - Mobile: `apps/mobile/lib/i18n.tsx` → add to the `translations` map.

## Adding a new locale (e.g. French)

1. Add the locale code to `packages/i18n/src/config.ts`:
   ```ts
   export const locales = ['nl', 'en', 'fr'] as const
   export const localeMeta = {
     ...,
     fr: { name: 'Français', flag: '🇫🇷', htmlLang: 'fr-FR' },
   }
   ```
2. Copy the directory: `cp -r packages/i18n/messages/en packages/i18n/messages/fr`.
3. Translate every value (keep keys identical — they're the contract).
4. Add French imports to `packages/i18n/src/index.ts` (one line per namespace).
5. Register `fr` in `apps/web/src/i18n/request.ts` and `apps/mobile/lib/i18n.tsx` — add a third entry to the message map per namespace.
6. That's it. The switchers auto-pick up the new entry from `locales`.

## Translating a new screen

Mechanical process — follow this every time:

1. **Extract:** read the screen, collect every user-facing Dutch string. Skip brand names ("Kliekjesclub"), icons, technical IDs.
2. **Name keys semantically, not literally.** `profile.sections.newDish`, not `profile.nieuw_gerecht`. Keys must be stable across languages — don't embed Dutch words.
3. **Group by namespace** — one JSON file per screen or domain (`profile.json`, `browse.json`, `reservations.json`, `checkout.json`).
4. **Write NL first, EN second.** Always keep key shapes identical between locales.
5. **Replace Dutch strings** in the component with `t('…')` calls.
6. **Test both locales** — use the switcher in the Footer (web) or Profile tab (mobile).

See `apps/web/src/components/Footer.tsx` + `packages/i18n/messages/{nl,en}/footer.json` for a full reference.

## What's NOT yet translated

- **Privacy policy** (`/privacy`, `/terms`) — long-form legal text stays Dutch-only until legal reviews the English translation. Pages themselves render content from `app/(consumer)/privacy/page.tsx` / `terms/page.tsx`.
- **CMS-backed content** in Supabase `page_content` rows — the code now falls back to locale-aware defaults, but rows stored in the database are still Dutch-only. Next step: add a `locale` column or a `page_content_translations` table so the CMS can store EN alongside NL.
- **Supabase auth emails** (sign-up, password reset, magic link) — configure extra templates in the Supabase dashboard → Authentication → Email Templates. Send the right one based on a future `profiles.locale` column.
- **Push notifications:** copy in `apps/mobile/lib/notifications.ts`.
- **Format helpers** (`lib/format.ts` — `formatPickupTime`, `formatRelativeDate`, `allergenLabel`) — still return Dutch strings. Most call sites now pass through locale-aware contexts, but the helpers themselves should take a locale parameter.
- **SEO**: `alternates.languages` / `hreflang` tags — will be added when the URL-prefix routing migration happens.

## Planned migration: URL-prefix locale routing (`/en/browse`)

The current web setup uses **cookie-based** locale detection — URLs don't change when you switch language. That's fine for phase 1 but suboptimal for SEO: Google can't index the English version separately.

When we're ready (probably once 80%+ of strings are extracted):

1. Move everything in `apps/web/src/app/` (except `api/`, `auth/`, `robots.ts`, `sitemap.ts`, `globals.css`) into `apps/web/src/app/[locale]/`.
2. Add `apps/web/middleware.ts`:
   ```ts
   import createMiddleware from 'next-intl/middleware'
   import { locales, defaultLocale } from '@kliekjesclub/i18n'

   export default createMiddleware({
     locales: [...locales],
     defaultLocale,
     localePrefix: 'as-needed', // or 'always' for /nl/ on Dutch too
   })

   export const config = {
     matcher: ['/((?!api|auth|_next|.*\\..*).*)'],
   }
   ```
3. Replace `cookies()`/`headers()` logic in `src/i18n/request.ts` with next-intl's `[locale]` param-aware config.
4. Add `alternates.languages` metadata + `hreflang` tags in root layout.
5. Update the language switcher to use `next-intl`'s routing helpers instead of the server action.

The catalog structure and `t()` call sites don't change. The migration is purely about routing + middleware.

## User preference persistence (roadmap)

Today the locale is stored:
- Web: in a `kc_locale` cookie (1 year).
- Mobile: in `expo-secure-store` under `kc_locale`.

For a logged-in user these won't sync across devices. When we add a `profiles.locale` column:
1. On login, read `profile.locale` and use it as the authoritative source.
2. When the user changes locale, write through to both the profile row and the local cookie/SecureStore.
3. On sign-up, store the detected locale so first-time emails go out in the right language.
